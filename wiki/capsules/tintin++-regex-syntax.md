# Tintin++ 正则语法陷阱

## 元信息

| 字段 | 值 |
|------|-----|
| 适用环境 | tt++ 2.02.61（所有平台） |
| 发现场景 | 为 mapper.tt 的 action 编写正则匹配中文 MUD 输出 |
| 验证状态 | ✅ 已通过自动化测试 + 实际 MUD 连接验证 |
| 最后更新 | 2026-04-14 |

## 一句话结论

tt++ 正则用 `{}` 分组不是 `()`，前导空格会被 strip 导致 `^    ` 永远匹配不到，GBK→UTF8 编码转换后行尾残留空格导致 `$` 锚定失效。`()` 在 action 正则是字面量不需转义。`#showme` 输出会被自己的 action 递归匹配。

## 关键发现

### 1. 分组语法：`{}` 不是 `()`

❌ `#action {^(.+) -$}` → 匹配不到
✅ `#action {^{.+} -$}` → 正确匹配

tt++ 使用花括号 `{}` 做正则分组和捕获，圆括号 `()` 不是分组语法。捕获组通过 `%1` `%2` ... 引用。

### 2. 前导空格被 strip

❌ `#action {^    这里明显的出口是 {.+}\.$}` → 永远不匹配
✅ `#action {^这里明显的出口是 {.+}\.$}` → 正确匹配

tt++ 在处理 session 接收到的数据时会去除行首空白（4个空格、tab 均会被 strip）。action 模式中不能依赖前导空格做匹配。

### 3. NPC 格式不固定

书剑 MUD 中 NPC 显示格式取决于显示模式（详见 `tintin++-mud-display-modes` 胶囊）：

❌ 只写 `#action {^◎{.+} \({.+}\)$}` → brief 模式下 NPC 捕获不到
✅ 同时写两个 action：
```
#action {^  {.+} \({.+}\)$} { ... }   ; brief 模式（两空格缩进，无 ◎）
#action {^◎{.+} \({.+}\)$} { ... }    ; 完整模式（有 ◎ 前缀）
```

### 4. 特殊字符转义（仅在 shell 生成 tt 文件时）

在 shell heredoc 或 `cat > file` 中写 tt++ 脚本时：

| 目标字符 | tt++ 原始写法 | shell heredoc 中写法 |
|----------|--------------|---------------------|
| `.`（字面量） | `\.` | `\\.` |
| `$`（行尾锚点） | `$` | `\$` |
| `(` `)`（字面量） | `\(` `\)` | `\\(` `\\)` |
| `{` `}`（字面量） | `\{` `\}` | `\\{` `\\}` |

**注意**：如果用文件编辑工具（非 shell）直接写 .tt 文件，不需要双重转义。

### 5. `%s` 可匹配空格

如果确实需要匹配可选的空白，可用 `%s`：

`#action {^%s这里明显的出口是 {.+}\.$}` → 能匹配（但捕获组行为需额外注意）

### 6. `+` 和 `*` 量词在 `{}` 内可用

- `{.+}` → 匹配一个或多个任意字符
- `{.*}` → 匹配零或多个任意字符
- `{[0-9]+}` → 匹配数字

### 7. 多分组 action 中 `%1` `%2` 的对应关系

`^{.+} - {.+}$` 中：
- `%1` = 第一个 `{.+}`（房间名，即 `-` 前的部分）
- `%2` = 第二个 `{.+}`（出口，即 `-` 后的部分）

### 8. `$` 锚定在 GBK→UTF8 模式下不可靠

`#config charset GBK1TOUTF8` 转换后，行尾会残留一个空格（0x20），导致 `$` 前面不是预期字符。

❌ `#action {^{.+} -$}` → 不匹配（`-` 后有空格）
✅ `#action {^{.+} -%s}` → 匹配（`%s` 吞掉尾部空格）

详见 `tintin++-gbk-encoding-pitfalls` 胶囊。

### 9. `()` 是字面量，不需要 `\(` 转义

在 tt++ action 正则中 `()` 不是分组（`{}` 才是），所以 `()` 直接匹配字面括号。NPC 匹配的正确写法：

❌ `#action {^  {.+} \({.+}\)$}` → 语法可能异常
✅ `#action {^  {.+}({.+})$}` → 正确匹配 `  当铺老板(Lao ban)`

详见 `tintin++-gbk-encoding-pitfalls` 胶囊。

### 10. `#showme` 输出会被自己的 action 递归匹配

`#showme <158>[地图] 房间: xxx` 这行输出会被 `^{.+} - {.+}$` 等 action 重新匹配，导致：
- 垃圾数据被写入 raw 文件
- 变量被意外覆盖

必须用 `map_active` flag 或其他机制隔离。详见 `tintin++-action-state-management` 胶囊。

## 已验证的正确方案

同时支持 brief 和完整模式的 action 集合（2026-04-14 更新）：

```tt
; brief 模式：房间名和出口在同一行
; 始终记录 temp_room，map_active 后才保存
#action {^{.+} - {.+}$} {
    #variable {temp_room} {%1};
    #if {"$map_active" == "1"} {
        m_flush_npcs {$current_room};
        #variable {current_room} {%1};
        #variable {current_room_exits_brief} {%2};
        m_autosave
    }
}

; 完整模式：房间名单独一行
; 注意 -%s 而不是 -$，GBK→UTF8 转换后行尾有空格
#action {^{.+} -%s} {
    #variable {temp_room} {%1}
}

; 完整模式：出口单独一行
#action {^这里明显的出口是 {.+}\.$} {
    #variable {current_room_exits} {%1}
}

; brief 模式：NPC — () 是字面量，不需要 \( 转义
#action {^  {.+}({.+})$} {
    #if {"$map_active" == "1"} {
        #list {current_npcs} {add} {%1(%2)}
    }
}

; 完整模式：NPC 有 ◎
#action {^◎{.+} ({.+})$} {
    #if {"$map_active" == "1"} {
        #list {current_npcs} {add} {%1(%2)}
    }
}
```

## 踩过的坑

| 坑 | 耗时 | 表现 |
|----|------|------|
| 用 `(.+)` 做分组 | ~2h | action 完全不触发，变量保持 NOT_SET |
| `^    ` 匹配前导空格 | ~1h | 出口行永远匹配不到，去掉前导空格后立即匹配 |
| shell heredoc 中 `\.` 只写一层 | ~30min | tt++ 解析为 `.`（任意字符），匹配过于宽泛 |
| NPC 假设都有 ◎ 前缀 | ~1h | brief 模式下 NPC 是两空格缩进无 ◎ |
| action 顺序错误（完整模式在前） | ~30min | `^{.+} -$` 先匹配 brief 数据，出口丢失 |
| `^{.+} -$` 不匹配（行尾空格） | ~3h | 完整模式 action 从未触发，GBK→UTF8 残留 0x20 |
| NPC 正则用 `\(` 转义括号 | ~2h | `()` 在 tt++ 本就是字面量，转义反而出错 |
| `#showme` 输出被 action 递归匹配 | ~1h | 登录垃圾数据写入 raw，变量被意外覆盖 |
